# WireGuard peer management and OpenVPN config refactor

This change adds WireGuard as a supported VPN protocol across the stack: a Curve25519 key generation package in the panel, a database migration for peer storage, node agent task actions for live peer management (`add_peer`, `remove_peer`, `sync_config`), and installer updates for `wireguard-tools` plus logrotate deployment. Alongside this, `applyOpenVPNServerConfig()` is refactored from line-by-line config manipulation to a single `TemplateEngine.Apply()` call.

Watch for: the `wireguard.add_peer` action does not validate public key format before passing it to `exec.Command` and writing it to the config file (confirmed). `wg syncconf` is passed the raw `wg0.conf` which will fail if it contains wg-quick directives like `Address` or `DNS` (likely). The `allowed_ips` field is also unvalidated — newlines in that value would corrupt the config file (confirmed).

## High-level view

The OpenVPN refactor collapses 40+ lines of ad-hoc config rewriting into a single `TemplateEngine.Apply()` call. Network validation that was explicitly called at the top of the old function is now performed inside `Apply()`, preserving behavior but introducing a subtle edge case when `OpenVPNNetwork` is empty (the old code would error; the new code skips validation silently).

WireGuard key generation is straightforward Curve25519 with proper clamping and good test coverage. The migration adds a `wg_peers` table with `private_key_encrypted TEXT NULL` — the encryption mechanism isn't part of this diff.

The node agent gains three WireGuard task actions that modify both the live interface (via `wg set`) and the persistent config (via file I/O on `/etc/wireguard/wg0.conf`). The main risk is in `add_peer`: no validation on public key format or `allowed_ips` CIDR format means malformed values get written to config and passed to system commands. The `remove_peer` config parser has a state machine that accumulates blank lines over repeated add/remove cycles.

<summary>Issues (5)</summary>

1. **No public key validation in add_peer** — `pubKey` is taken from the payload and passed directly to `exec.Command` and written to config without format validation. Add a base64 length/charset check (44 chars, valid base64, decodes to 32 bytes) before use.
2. **Preshared key written plaintext to wg0.conf** — The PSK is appended in cleartext to the config file on disk. If this is intentional (WireGuard itself requires it), document the trust model; if not, consider using `wg set` with `/dev/stdin` only for runtime without persisting to the file.
3. **syncconf passed unstripped config** — `wg syncconf wg0 /etc/wireguard/wg0.conf` will fail if the config contains `Address`, `DNS`, `Table`, or other wg-quick directives. Use `wg-quick strip wg0` to produce the stripped version first.
4. **No allowed_ips format validation** — The `allowed_ips` value is written to config and passed to `wg set` without CIDR format validation. A value containing newlines would corrupt the config file.
5. **removePeerFromConfig trailing newline accumulation** — Each `add_peer` appends `\n[Peer]\n...` which adds blank lines. Repeated add/remove cycles accumulate empty lines since removal only strips trailing blanks immediately before the matched `[Peer]` header.

## Peer input validation gap

The `wireguard.add_peer` handler validates that `public_key` and `allowed_ips` are non-empty and non-nil, but performs no format validation. A WireGuard public key is exactly 44 base64 characters decoding to 32 bytes. Without this check:

```go
pubKey := fmt.Sprint(payload["public_key"])
// ... passed directly to:
cmd := exec.Command("wg", "set", "wg0", "peer", pubKey, ...)
```

While `exec.Command` separates arguments (so shell injection isn't possible), a malformed key will produce a confusing error from `wg` after the config file has already been appended to in some code paths. More critically, `allowed_ips` values containing newlines would inject arbitrary lines into `/etc/wireguard/wg0.conf` because the value is interpolated directly into `fmt.Sprintf`:

```go
peerBlock := fmt.Sprintf("\n[Peer]\nPublicKey = %s\nAllowedIPs = %s\n", pubKey, allowedIPs)
```

The existing `vpn.disconnect-user` handler validates usernames with `isValidVPNUsername()`; an analogous validator is warranted here. **Confidence: confirmed** — no validation exists in the diff.

## syncconf and wg-quick directives

`wg syncconf` accepts only the subset of config that `wg` understands — it rejects wg-quick-specific directives like `Address`, `DNS`, `PostUp`, `PostDown`, `Table`, `MTU`, `SaveConfig`. If `wg0.conf` contains any of these (the norm for a wg-quick-managed interface), `wg syncconf wg0 /etc/wireguard/wg0.conf` will fail.

The FEAT-001 spec originally mentioned "run `wg-quick strip wg0` to get runtime config, compare with file, run `wg syncconf wg0` if needed" but the implementation skipped the strip step:

```go
case "wireguard.sync_config":
    cmd := exec.Command("wg", "syncconf", "wg0", "/etc/wireguard/wg0.conf")
```

The fix is to pipe `wg-quick strip wg0` output to a temp file and pass that to `syncconf`. **Confidence: likely** — any standard wg-quick config will have `Address` at minimum.

## Preshared key on disk

When a PSK is provided, it's written in plaintext to `/etc/wireguard/wg0.conf`:

```go
peerBlock = fmt.Sprintf("\n[Peer]\nPublicKey = %s\nPresharedKey = %s\nAllowedIPs = %s\n", pubKey, psk, allowedIPs)
```

The file is opened with mode 0600 (root-only), which matches WireGuard convention. But the panel also stores PSKs in the database (`preshared_key VARCHAR(44) NULL`). Two copies of the same secret increases the attack surface — a compromise of either the node filesystem or a database dump leaks the PSK. **Confidence: confirmed** — the code writes PSK to disk and the migration stores it server-side.

## removePeerFromConfig state machine

The parser tentatively appends `[Peer]` lines to `result`, then backtracks if the next `PublicKey` line matches. The trailing-empty-line trimming (`for len(result) > 0 && strings.TrimSpace(result[len(result)-1]) == ""`) could over-trim blank lines between unrelated sections if multiple blanks separate them.

After repeated add/remove cycles, blank lines accumulate because `add_peer` always prepends `\n` before `[Peer]` but removal only strips blanks immediately before the matched header, not after the removed block. **Confidence: confirmed** — the append format starts with `\n`.

## OpenVPN refactor: empty network edge case

The old code called `ValidatePrivateNetwork(v.OpenVPNNetwork, false)` unconditionally. The new code passes `v.OpenVPNNetwork` as `vars.Network`, and `engine.Apply()` only validates when `vars.Network != ""`. If `OpenVPNNetwork` is ever empty, the old code would return a validation error while the new code proceeds silently, rendering a config without a `server` directive. **Confidence: possible** — depends on whether empty-network calls to this function are reachable.
