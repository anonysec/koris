# WireGuard peer management and OpenVPN config refactor (v2)

This change adds WireGuard as a supported VPN protocol across the stack: Curve25519 key generation in the panel, a database migration for peer storage, node agent task actions for live peer management (`add_peer`, `remove_peer`, `sync_config`), installer updates for `wireguard-tools` and logrotate, and a refactor of `applyOpenVPNServerConfig()` to use `TemplateEngine.Apply()`. The v1 review raised five issues — all have been addressed in the follow-up commit.

Watch for: `defer os.Remove(tmpPath)` in `sync_config` fires on `executeTask` return rather than at case-block scope — functionally correct since the function returns immediately, but fragile if the function grows (possible).

**Verdict**: APPROVED

## High-level view

All five v1 issues are resolved. `isValidWireGuardKey` validates 44-char base64 decoding to 32 bytes, called before any `exec.Command` or file write. `isValidAllowedIPs` rejects newlines and requires every comma-separated segment to pass `net.ParseCIDR`. `sync_config` now pipes `wg-quick strip wg0` through a temp file before calling `wg syncconf`. The OpenVPN refactor adds an explicit `v.OpenVPNNetwork == ""` guard before reaching `TemplateEngine.Apply()`. `removePeerFromConfig` trims trailing empty lines at the end of processing, and the `add_peer` config append checks whether the file already ends with a newline before prepending one.

Unit tests cover key validation (valid keys, short keys, invalid base64, embedded newlines), allowed_ips validation (CIDRs, injection attempts, empty segments), and `removePeerFromConfig` (middle peer removal, last peer removal, trailing newline assertions).

## add_peer atomicity gap

If `wg set` succeeds but the subsequent config file write fails (disk full, permission change), the peer exists on the live interface but not in the persistent config. A reboot or `sync_config` call would lose the peer, since `sync_config` reconciles file-to-runtime only (it syncs the file *into* the runtime, not vice versa). The panel receives the error response and can retry, and this is standard for WireGuard management tools, but the failure mode is fail-open-at-runtime-then-lose-on-restart rather than fail-closed. (likely)
