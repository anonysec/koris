# WireGuard API, bandwidth collector, cert rotation, and IPv6 tc rules

This change adds four capabilities across the VPN panel and node agent: WireGuard API endpoints for peer CRUD and client config download, a real-time per-user bandwidth display backed by tc class statistics, an automatic certificate rotation worker with node distribution, and IPv6 dual-stack tc filter rules in the OpenVPN connect script. The node agent gains `cert.distribute` and `wireguard.sync_config` task actions, and the push payload is extended with `PublicIPv6` and `PerUserBandwidth` fields.

Watch for: the panel stores WireGuard private keys in plaintext in the database (confirmed). The `createWireguardPeer` endpoint does not validate `AllowedIPs` format before writing to the database and dispatching the task to the node (confirmed). The cert rotation worker blindly regenerates and distributes certificates to all online nodes based on filename heuristics, with no rollback mechanism if distribution partially fails (confirmed). The bandwidth collector conflates RxBytes and TxBytes into the same value from a single tc class counter (confirmed).

## High-level view

The WireGuard API endpoints generate key pairs server-side and store the raw private key in a column named `private_key_encrypted` without actually encrypting it. The client config download endpoint then serves that key verbatim. The peer creation flow dispatches a task to the node agent, which has solid input validation (`isValidWireGuardKey`, `isValidAllowedIPs`), but the panel-side endpoint skips that validation before writing to the DB.

The bandwidth collector parses `tc -s class show dev` output into per-class byte counters and computes delta rates between polls. The design maps a single `Sent` counter to both `RxBytes` and `TxBytes` — this conflation means the panel's `rx_bps` and `tx_bps` columns always store identical values per user. The IP derivation from class ID minor number is fragile (only the last octet, matched against radacct's framed IP last octet).

The cert rotation worker runs hourly, queries `vpn_certificates` for anything expiring within 30 days, and auto-regenerates when within 7 days. The `certType()` classifier relies on filename substring matching (`contains "ca"` → CA, `contains "server"` → server), which can misclassify ambiguous paths. Regeneration is self-signed for both CA and server certs without CSR/signing-chain preservation. Distribution creates `cert.distribute` tasks for all online/stale nodes — there's no idempotency check, so if the worker fires while previous distribution tasks are still pending, duplicate tasks accumulate.

The IPv6 tc rules in `koris-client-connect.sh` are correctly guarded by `[ -n "$IPV6" ]` and apply higher priority numbers (prio+10000) to avoid conflicts with IPv4 rules.

<summary>Issues (8)</summary>

1. **Private key stored unencrypted** — `private_key_encrypted` column receives the raw base64 private key from `GenerateKeyPair()`. Implement actual encryption (AES-GCM with a panel-level master key) before storing, or document the trust model explicitly.
2. **No AllowedIPs validation in panel API** — `createWireguardPeer` checks `in.AllowedIPs == ""` but doesn't validate CIDR format. A malformed value gets stored in the DB and dispatched as a task payload. Add the same `net.ParseCIDR` validation used on the node side.
3. **Rx/Tx conflation in bandwidth collector** — `DeltaRates` assigns the same `cs.Bytes` to both `RxBytes` and `TxBytes`, and the same computed rate to both `RxBps` and `TxBps`. Either collect from two different tc devs (egress on tun0 + ingress qdisc) or rename to a single `Bytes`/`Bps` field to avoid misleading the frontend.
4. **certType filename heuristic misclassification** — A file named `/etc/openvpn/server/cascade-vpn.crt` would match "ca" (substring of "cascade"). Use `filepath.Base` with stricter matching (exact name or prefix/suffix patterns).
5. **No duplicate task suppression in cert distribution** — If the worker fires again before prior `cert.distribute` tasks complete, identical tasks accumulate in `node_tasks`. Add a check: skip nodes that already have a pending `cert.distribute` task for the same `cert_path`.
6. **Self-signed CA regeneration breaks existing clients** — `Regenerate` creates a new self-signed CA with a fresh key pair. All existing client certs signed by the old CA become untrusted. The worker should emit a critical event and require manual approval for CA rotation instead of auto-regenerating.
7. **certWorker.Stop() never called** — The panel `main.go` starts the cert worker but has no shutdown hook. The goroutine leaks on process exit (minor, as OS reclaims), but more importantly, a rotation in progress at shutdown could leave a half-written cert on disk.
8. **IP-to-username matching by last octet** — The bandwidth payload joins `bw.IP` (which is the tc class minor number, e.g. "52") against `radacct.framedipaddress` last octet. If two users on different nodes have the same last octet, or if the tc class minor doesn't correspond to the IP last octet (custom tc setup), the mapping is wrong. Document this assumption or carry the full IP from tc filter metadata.

## Private key handling in the WireGuard API

The `createWireguardPeer` handler calls `wireguard.GenerateKeyPair()` and stores the raw private key:

```go
res, err := s.DB.Exec(`
    INSERT INTO wg_peers (customer_id, node_id, public_key, preshared_key, private_key_encrypted, allowed_ips, status)
    VALUES (?, ?, ?, ?, ?, ?, 'active')`,
    in.CustomerID, in.NodeID, publicKey, presharedKey, privateKey, in.AllowedIPs)
```

The column is named `private_key_encrypted` but receives the unencrypted base64 string. The `wireguardPeerConfig` endpoint then reads it back and passes it directly to `GenerateClientConfig` as the client's `PrivateKey`. A database dump or SQL injection elsewhere in the application exposes every client's private key. **Confidence: confirmed** — no encryption call exists between `GenerateKeyPair()` and `DB.Exec`.

The preshared key is also stored in plaintext in the same row. Both values are included in the task payload sent to the node agent, traveling over HTTP (the panel URL is configurable and may not use TLS in all deployments).

## Panel-side AllowedIPs validation gap

The node agent has a thorough `isValidAllowedIPs` function that rejects newlines and validates each comma-separated segment with `net.ParseCIDR`. But the panel's `createWireguardPeer` only checks for empty:

```go
if in.NodeID <= 0 || in.AllowedIPs == "" {
    writeJSONCode(w, http.StatusBadRequest, ...)
    return
}
```

An attacker with panel access (admin-only endpoint, but still) could store arbitrary strings in the database. The node agent will reject them at execution time, but the task will fail silently — the peer record stays "active" in the DB without actually being configured on the node. **Confidence: confirmed** — no format validation in the panel endpoint.

## Bandwidth collector Rx/Tx conflation

In `bandwidth.go`, `DeltaRates` assigns:

```go
bw := UserBandwidth{
    ClassID: cs.ClassID,
    RxBytes: cs.Bytes,
    TxBytes: cs.Bytes,  // same value
}
// ...
bw.RxBps = rate
bw.TxBps = rate  // same rate
```

The tc class `Sent` counter on the tun device represents bytes transmitted through that class — which is downstream (panel→client) traffic on the tun device. This is only the download direction for the user. Upload traffic would need a separate ingress qdisc or a second device's stats. By assigning the same value to both Rx and Tx, the frontend displays identical upload/download numbers. **Confidence: confirmed** — the code assigns `cs.Bytes` to both fields.

## certType classifier fragility

```go
func certType(path string) string {
    base := strings.ToLower(filepath.Base(path))
    if strings.Contains(base, "ca") {
        return "ca"
    }
    // ...
}
```

This fires before the "server" check, so a file named `local-ca-signed-server.crt` would be classified as "ca" even if it's a server cert. More critically, a path like `/etc/openvpn/server/vpn-cascade.crt` matches because "cascade" contains "ca". The ordering means "ca" takes priority over "server" even when both substrings appear. **Confidence: confirmed** — `strings.Contains("cascade", "ca")` returns true.

## Certificate rotation self-signed CA hazard

When `DaysUntilExpiry <= 7` and `certType` returns "ca", the worker auto-regenerates:

```go
cmd = exec.Command("openssl", "req", "-x509", "-nodes",
    "-days", "3650",
    "-newkey", "ec",
    "-pkeyopt", "ec_paramgen_curve:prime256v1",
    "-keyout", keyPath,
    "-out", cert.CertPath,
    "-subj", "/CN=VPN-CA")
```

This generates a brand-new key pair. All previously issued client certificates are now invalid because they were signed by the old CA's key. The `DistributeToNodes` call pushes the new CA cert to nodes, but connected clients with the old CA cert will fail to verify. There's no CRL update, no client re-provisioning step, and no warning to the admin beyond the event log entry. **Confidence: confirmed** — `-newkey` generates a fresh private key.

## Bandwidth snapshot storage unbounded growth

Every push cycle (default 10 seconds per node), each node inserts one row per active tc class into `user_bandwidth_snapshots`:

```go
_, _ = s.DB.Exec(`INSERT INTO user_bandwidth_snapshots(node_id, username, ip, rx_bps, tx_bps) VALUES(?,?,?,?,?)`,
    nodeID, username, bw.IP, bw.RxBps, bw.TxBps)
```

With 100 nodes, 50 active users per node, and 10-second intervals: 100 × 50 × 6/min × 60 × 24 = 43.2M rows/day. The `bandwidthPayload()` query uses `WHERE created_at >= NOW() - INTERVAL 30 SECOND` which requires an index on `created_at` to stay performant at scale. No cleanup/TTL mechanism is visible in this diff. **Confidence: likely** — no DELETE or partition scheme is shown; the migration 023 is a no-op `SELECT 1`.
