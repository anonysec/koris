# Admin and portal UI enhancements

This change adds several missing features to the admin and portal interfaces: an i18n language switcher in the admin topbar, a node delete capability (frontend + backend), conditional VPN protocol config forms (PSK visibility based on auth mode), payment management via KDrawer overlays, a dashboard chart that shows cumulative data usage instead of speed, and a simplified portal navigation with a user dropdown menu.

Watch for: The cumulative "Data Usage" chart sums raw bps values without converting to bytes, producing misleading totals (confirmed). The `deleteNode` backend handler silently discards errors from related-table cleanup, which can leave orphaned data if only the final `DELETE FROM nodes` succeeds (confirmed). PSK values persist in `extra_json` when the user switches away from IPSec/PSK mode, sending stale secrets to the backend (confirmed).

## High-level view

The i18n consolidation moves inline translation registration out of `main.ts` into the dedicated `i18n.ts` module, where `registerMessages` is called as a side-effect import. The topbar gains a three-button language switcher wired to the shared composable — but the local `i18n.ts` still exports its own `locale` ref and `setLocale` that are now redundant with the shared composable's state, creating two sources of truth.

The node delete flow is protected by `requireAdmin` middleware and uses a confirmation dialog before calling `DELETE /api/nodes/:id`. The backend performs cascade cleanup of related tables but silently discards errors from those DELETEs. Tables referenced in `fillNodeRuntime` (`node_services`, `node_usage_snapshots`, `node_diagnostics`) aren't cleaned up at all.

The VPN protocol config forms gain conditional PSK visibility for L2TP (IPSec mode toggle) and IKEv2 (auth_type selector), plus advanced settings (MTU, max_clients, conn_limit, enable_logs) in a collapsible `<details>` block. PSK values aren't cleared from form state when the user switches modes, so stale secrets persist in the saved config JSON.

The dashboard chart conversion from speed to "data usage" has a unit error: `rxHistory` contains bps readings pushed every ~3 seconds, and the computed sums them without multiplying by the polling interval or dividing by 8. The fallback display (total from live sessions via `input_bytes`/`output_bytes`) is correct.

The portal navigation reduces from 6 to 4 tabs and moves Profile behind a user dropdown. The `portal-usage` route still exists in the router but has no navigation entry point after the tab removal.

## Dashboard data-usage chart miscalculation

The realtime store pushes `total_rx_bps` and `total_tx_bps` into `rxHistory`/`txHistory` arrays every ~3 seconds via `scheduleHistoryUpdate(d.total_rx_bps || 0, d.total_tx_bps || 0)`. The new `trafficChartData` computed accumulates these:

```typescript
let cumulative = 0
return realtime.rxHistory.map((rx, i) => {
  cumulative += rx + (realtime.txHistory[i] || 0)
  return { label: `${i}`, value: cumulative }
})
```

Because `rx` is bits-per-second, the cumulative sum represents "total bps samples summed" — not bytes transferred. To approximate actual bytes, each sample should be `(bps × interval_seconds) / 8`. Without this conversion, a user seeing "4,500,000" on the Y-axis would interpret it as ~4.5 MB, but it's actually a unitless sum of speed readings.

The fallback display correctly pulls `input_bytes`/`output_bytes` from live sessions, so the issue is specific to the chart path.

## Node deletion cascade and error handling

```go
_, _ = s.DB.Exec(`DELETE FROM node_vpn_configs WHERE node_id=?`, id)
_, _ = s.DB.Exec(`DELETE FROM node_tasks WHERE node_id=?`, id)
_, _ = s.DB.Exec(`DELETE FROM node_status WHERE node_id=?`, id)

if _, err := s.DB.Exec(`DELETE FROM nodes WHERE id=?`, id); err != nil {
```

Three problems compound here:

1. All cleanup DELETEs discard their errors — a failed cleanup is invisible.
2. The `fillNodeRuntime` function queries `node_services`, `node_usage_snapshots`, and `node_diagnostics` for each node, but `deleteNode` doesn't clean those tables. Orphaned rows accumulate.
3. No transaction wraps the sequence — if the final `DELETE FROM nodes` succeeds but an earlier cleanup failed, the system is in an inconsistent state with no way to retry.

## PSK persistence across mode switches

In the L2TP form, switching from 'ipsec' to 'plain' hides the PSK field via `v-if="configForm.extra_json.ipsec_mode === 'ipsec'"`, but `configForm.extra_json.psk` retains whatever the user typed. When the form is saved, the payload includes the stale PSK in `extra_json`. The backend stores it verbatim. The same pattern affects IKEv2 when switching from 'psk' to 'certificate'.

This is a credential hygiene issue: a PSK intended for one configuration persists in the database even when the protocol mode no longer uses it. Add a watcher or `@change` handler to clear `psk` when the mode changes away from PSK-based auth.

## Dual locale state

After this change, two locale systems coexist:

- `panel/web/admin/src/i18n.ts` exports `locale` (a Vue `ref`) and `setLocale(l: Locale)`
- `TheTopbar.vue` imports `useI18n()` from `@koris/composables/useI18n` and calls its `setLocale`

The `registerMessages(messages)` call correctly feeds translations into the shared system, but the local `t()` function reads from the local `locale` ref. When the topbar switches language via the shared composable, the local ref remains unchanged. Any component using `import { t } from '@/i18n'` will continue rendering the original locale.
