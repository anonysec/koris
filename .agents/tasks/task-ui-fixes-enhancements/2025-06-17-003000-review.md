# UI fixes and admin panel enhancements (v2)

All six issues from the v1 review have been addressed. The notification bell now routes to tickets and calls `markAllRead()`. VPN Cores labels (~35) are fully wrapped in `t()` with EN/FA/ZH translations. Telegram form is wired with `v-model`, loads on mount, and saves via PATCH. Import backup is labeled "(coming soon)" in all locales. Confirm dialog and token banner use `t()`.

**Watch for:** Toast messages and export item labels remain hardcoded English, escaping the i18n system (confirmed). Export via `window.open()` will 401 if admin auth is header-only (likely). The "Get this from @BotFather" hint string is hardcoded (confirmed).

**Verdict**: APPROVED — the remaining issues are polish-level and don't block shipping. The six blocking concerns from v1 are all resolved.

## High-level view

The i18n pass is thorough for template-rendered text: every `KFormField` label, section title, button, empty state, and confirm dialog now routes through `t()`. The gap is in script-level strings — toast notifications ("Panel settings saved successfully", "Configuration saved", etc.) and the `exportItems` array labels ("Customers CSV", "Payments CSV") stay English regardless of locale. This is a second-tier concern since toast messages are transient and export labels are semi-technical, but it does mean the panel isn't fully multi-language at the moment.

The Telegram form is now functional with `loadTelegramSettings` pulling from the same `/api/panel-settings` endpoint as panel settings. Both share a single GET call on mount via `Promise.all`, which is efficient. The form saves via PATCH, which merges correctly with other panel settings keys.

Settings view is lean: five tabs, no dead UI, responsive breakpoints for mobile via `@media (max-width: 768px)`. The topbar search box collapses to icon-only on `<640px`. VPN Cores protocol cards stack vertically on mobile with controls realigning. Mobile-friendliness is addressed at the CSS level for the views touched in this diff.

## Residual i18n gaps in script-level strings

The template layer is fully translated — every visible label passes through `t()`. But seven toast calls across SettingsView and one in NodesView use inline English:

```ts
toast.success('Panel settings saved successfully.')
toast.error('Failed to save panel settings.')
toast.success('Data warning thresholds saved successfully.')
toast.error('Failed to save data warning thresholds.')
toast.success('Telegram settings saved successfully.')
toast.error('Failed to save Telegram settings.')
toast.success('Import functionality coming soon.')
// NodesView:
toast.success('Configuration saved')
```

The `exportItems` array is also defined with string literals:

```ts
const exportItems: ExportItem[] = [
  { label: 'Customers CSV', url: '/api/export/customers.csv' },
  { label: 'Payments CSV', url: '/api/export/payments.csv' },
  ...
]
```

Since these labels render inside the template (`{{ item.label }}`), they won't change when the locale switches. Converting `exportItems` to a `computed` that calls `t()` would close this gap.

Similarly, `hint="Get this from @BotFather"` on the Telegram token field is a raw English prop.

None of these are blocking — they're all quick additions to the existing i18n key map — but they prevent the panel from being fully multi-language as the user requested.

## Export download auth model

The backup section uses `window.open(url, '_blank')` for CSV downloads. The Go backend protects these routes with `requireAdmin` middleware. If admin authentication relies on `Authorization: Bearer <token>` headers (set by an Axios interceptor or similar), a raw browser navigation won't include that header, resulting in a 401 in the new tab. Cookie-based sessions would work fine.

If this is a concern in the deployed environment, the standard fix is to fetch the URL via the API client (which attaches headers), convert the response to a Blob, and trigger a download via `URL.createObjectURL`. This is a minor UX issue — the download simply won't start — not a security gap.
