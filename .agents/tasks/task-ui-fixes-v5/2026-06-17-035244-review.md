# Node edit, single-page portal, RTL, and five-theme system

This change delivers four features on the admin/portal frontend: inline node editing with live validation and config preview, a single scrollable portal page replacing the tab-based layout, comprehensive RTL support for the Farsi locale, and a full UI theme system with five themes (Midnight, Kiro, GitHub, Soft Dark, Corporate) plus system-default mode detection. Both admin and portal fetch the admin-set theme from the server on mount; the admin panel's settings page provides theme cards with live preview and persists the choice to `/api/panel-settings`.

**Watch for:** The portal fetches theme settings from `/api/panel-settings` without authentication gating — if that endpoint requires admin auth, portal users will always fall back to localStorage defaults (likely). The `toggle()` function in the legacy theme API switches mode from 'system' to 'dark' on first click because the else branch doesn't handle 'system' distinctly (confirmed). The node edit form sends all three fields on every save even when unchanged, creating potential for accidental overwrites on concurrent edits (confirmed).

## High-level view

The theme composable was rewritten from a simple dark/light toggle into a two-axis system: mode (system/dark/light) controls the color scheme, and theme (five palettes) controls the CSS variable set. Module-level singleton state means all consumers share the same reactive refs. The "Corporate" theme forces light mode regardless of the mode radio — but the mode radio doesn't disable or explain this, so an admin can select "Dark" with no visible effect.

The portal was flattened from a multi-route tab layout into a single `SinglePageView.vue` (776 lines) that stacks account status, usage gauge, VPN profiles, and support tickets vertically. The old routes redirect to `/`. The shell retains only a header with language switcher, theme toggle, and user menu — but no `@media` breakpoint handles small screens for this action row.

Node management gained an inline edit form (name, IP, domain) backed by a `PATCH /api/nodes/:id` store action, plus client-side validation for port and CIDR, a config preview panel, and chip-array inputs for push routes and SSH allowed keys.

## Theme propagation and the portal auth gap

The admin saves `ui_theme` and `ui_mode` to `/api/panel-settings` via PATCH, and both apps fetch from that endpoint on mount:

```typescript
const res = await get<{ ok: boolean; settings: Record<string, string> }>('/api/panel-settings')
```

If this endpoint is gated behind admin-level auth (typical for a settings management endpoint), the portal's request will 401, the catch swallows silently, and portal users see whatever their localStorage holds — which defaults to 'midnight' theme and 'system' mode. The admin's theme choice never reaches portal users unless the API explicitly allows portal-auth tokens to read panel settings, or a separate public endpoint is exposed.

## Legacy toggle behavior under three-state mode

```typescript
function toggle(): void {
  if (mode.value === 'dark') {
    setMode('light')
  } else {
    setMode('dark')
  }
}
```

When mode is 'system' (the default for new users per the user's request), the first click sets mode to 'dark' regardless of what the system preference actually is. If the user's OS is already in dark mode, the visual result is no change on first click, then a switch to light on second click. The portal header's moon/sun button calls this function directly. Fix: resolve the current effective mode and toggle to the opposite, or remove the toggle from portal since the admin controls the theme.

## Node edit: always-dirty PATCH payload

```typescript
const success = await store.editNode(editingNodeId.value, {
  name: editNodeForm.value.name,
  public_ip: editNodeForm.value.public_ip,
  domain: editNodeForm.value.domain,
})
```

If another admin changed the domain via a different session, this PATCH overwrites it with the stale value from this session's form load. The `EditNodePayload` interface already has all fields optional, confirming the backend supports partial updates — build the payload from only changed fields.

## Corporate theme mode radio inconsistency

When Corporate is selected, `resolveEffectiveMode` always returns 'light' via the `forcedMode` flag:

```typescript
if (themeInfo?.forcedMode) {
  return themeInfo.forcedMode
}
```

The mode radio in settings doesn't visually disable or display an explanation when Corporate is active. An admin can select "Dark" mode but see no effect. The mode section should show a disabled state or a note when the active theme forces a mode.
