# Deploy script fixes, dead code removal, and diagnostic cleanup

Three commits tidy the deploy pipeline and remove ~280 lines of dead Go code. The deploy scripts (`deploy.sh`, `deploy-report.sh`) had a broken variable reference (`PANEL_PORT`, never defined) that caused health checks to hit port 8088 instead of the configured address; this is replaced with `PANEL_ADDR` parsed from `/etc/panel/panel.env`, defaulting to `127.0.0.1:8080` (matching `config.go`). A duplicate health-check block in post-deploy diagnostics is collapsed into the single authoritative check that already gates deploy success. Three unreferenced internal packages (`gateway`, `totp`, `certrotation`) and one unused method (`logServerError`) are deleted.

**Watch for:** The `deploy-report.sh` invocation sources `/etc/panel/panel.env` into the main shell before backgrounding — if that file sets `set -e` or redefines variables used later by the parent shell, it could interfere (possible, needs env file inspection). The TOTP removal is safe today but removes a correct RFC 6238 implementation that may need to be re-written if 2FA is added later (informational, no action needed).

## High-level view

The deploy variable fix replaces `PANEL_PORT` (undefined, always falling through to `8088`) with `PANEL_ADDR` parsed from the env file, defaulting to `127.0.0.1:8080` to match the Go binary's config.

The dead-code removal is clean — all three packages have zero imports anywhere in the tree, `logServerError` has zero call sites, and the build passes after deletion.

The deploy-report invocation lost its subshell isolation: the `source` now runs in the parent shell. Since `deploy-report.sh` is the last command in the script this is harmless in practice, but a subshell wrapper would be strictly safer.

<summary>Issues (2)</summary>

1. **Sourcing panel.env in parent shell** — `deploy.sh` now sources `/etc/panel/panel.env` into the parent shell (previously it was inside a subshell). If that file contains `set -e`, traps, or variable overrides, they could theoretically affect subsequent shell behavior. Wrap the source+exec in a subshell: `( source /etc/panel/panel.env; exec bash ./deploy-report.sh ) &`. Confidence: possible.
2. **deploy-report.sh re-parses PANEL_ADDR independently** — Both `deploy.sh` and `deploy-report.sh` parse `PANEL_ADDR` from `/etc/panel/panel.env` with identical grep logic. If deploy.sh already sources that file before invoking the report script, the grep in deploy-report.sh is redundant and the two could drift. Consider having deploy-report.sh rely on the exported env var. Confidence: confirmed (both files contain the same grep+cut+default block).

## PANEL_ADDR duplication between scripts

Both scripts now contain the identical parsing block:

```bash
PANEL_ADDR="$(grep -E '^PANEL_ADDR=' /etc/panel/panel.env 2>/dev/null | cut -d= -f2 | tr -d "'"\" || true)"
PANEL_ADDR="${PANEL_ADDR:-127.0.0.1:8080}"
```

Since `deploy.sh` sources the env file before calling `deploy-report.sh`, the variable is already in the environment — the grep inside the report script is redundant. If someone later changes the default in one place but not the other, health checks in the report could target a different address than the one deploy.sh validated against.

## deploy-report.sh invocation isolation

Previously:
```bash
(source /opt/koris-next/panel.env 2>/dev/null; bash ./deploy-report.sh) 2>/dev/null &
```

Now:
```bash
if [ -f /etc/panel/panel.env ]; then
  source /etc/panel/panel.env
fi
bash ./deploy-report.sh &
```

The subshell isolation is gone — `source` now runs in the parent. If the env file contains traps or modifies shell options, the backgrounded child inherits that state. A subshell wrapper would be safer: `( source /etc/panel/panel.env 2>/dev/null; bash ./deploy-report.sh ) &`.

The path also changed from `/opt/koris-next/panel.env` to `/etc/panel/panel.env`, aligning with where the rest of the script expects config to live.
