# Cores tab grouped forms, language sync fix, customers filter layout, node installer hardening

This change addresses four user-reported issues in a single branch: the customers page filter tabs now sit beside the search input instead of stacking vertically; language switching from the sidebar now propagates to the settings page and vice versa via bidirectional watchers on the shared locale ref; the node installer gains connectivity pre-checks, Go version management, and post-install health verification; and the cores/nodes configuration form is reorganized from a flat field list into grouped sections (Networking, Security, Performance, Logging) with danger-field indicators and reset/restart buttons. A NotificationsView page and topbar dropdown are added as supporting work. All i18n strings are added across EN/FA/ZH for the new and modified views.

**Watch for:** The topbar notification dropdown's `formatTime` function uses untranslated English strings ("just now", "5m ago") which breaks the i18n promise for FA/ZH users (confirmed). The `resetToDefaults` function references `PROTOCOL_DEFAULTS` which must already exist in the file — if a protocol key is missing the form will silently fill with `undefined` values (likely). The node installer downloads Go binaries from `go.dev` without checksum verification (confirmed).

## High-level view

The language switching fix uses bidirectional watchers between `SettingsView`'s local `panelLang` and the global `currentLocale` singleton ref. `AppShell` wires the sidebar's `change-lang` event to `setLocale`. The gap: `TheTopbar`'s `formatTime` helper bakes English relative-time strings directly instead of using `t()`, and the language control now lives only in the sidebar (inaccessible on mobile without opening it).

The customers view layout is a CSS-structural change — status tabs and search now share a flex row that collapses on mobile. The resellers view adds `status` and `customer_count` columns that reference store fields which may not exist on older API responses (uses optional chaining and defaults to guard).

The cores tab replaces `<details>` collapsibles with semantic form groups and adds `isDangerField` for visual-only danger indicators. No confirmation dialog prevents saving a dangerous config (e.g., `tls_mode: none`). `resetToDefaults` will throw if `PROTOCOL_DEFAULTS` lacks a key for the active protocol.

The node installer validates panel reachability before proceeding, manages Go installation with version detection, and runs post-install health checks. Go binaries are fetched without SHA256 verification, and architecture detection uses `dpkg --print-architecture` which doesn't exist on RPM distros — the `|| echo amd64` fallback breaks on ARM64 CentOS/Rocky systems.

## Untranslated relative timestamps in the topbar notification dropdown

The `formatTime` function in `TheTopbar.vue` returns hardcoded English strings:

```ts
if (minutes < 1) return 'just now'
if (minutes < 60) return `${minutes}m ago`
const hours = Math.floor(minutes / 60)
if (hours < 24) return `${hours}h ago`
const days = Math.floor(hours / 24)
return `${days}d ago`
```

The same function is duplicated in `NotificationsView.vue` but there it uses `toLocaleString()` which adapts to browser locale. The topbar version — visible on every page — will show English time strings to FA and ZH users.

## Go binary integrity and architecture detection in node-install.sh

The installer fetches Go without verification:

```bash
curl -fsSL "$GO_URL" | tar -C /usr/local -xz
```

A compromised CDN or MITM could inject a tampered Go toolchain that then compiles the node agent. Adding `sha256sum` verification against the published checksums at `go.dev/dl/` would close this.

The architecture detection uses `dpkg --print-architecture 2>/dev/null || echo amd64`. On RPM-based systems (CentOS, Rocky, AlmaLinux), `dpkg` doesn't exist, so the fallback triggers. This works on x86_64 but on ARM64 servers the script downloads the wrong binary. A portable fix:

```bash
case "$(uname -m)" in
  x86_64) ARCH="amd64" ;;
  aarch64) ARCH="arm64" ;;
  *) ARCH="amd64" ;;
esac
```

## Language switching: mobile accessibility gap

The language switcher was removed from `TheTopbar.vue` (replaced by the notification bell) and now exists only in `TheSidebar.vue`. On mobile, the sidebar is collapsed and there's no language control in the bottom tab bar or mobile hamburger menu. Users on mobile would need to explicitly open the sidebar to switch language.

## Cores tab: danger indicators without confirmation

The `isDangerField` function flags combinations like OpenVPN `tls_mode: none` and L2TP `ipsec_mode: plain` with a red left-border visual cue. However, the implementation is purely cosmetic — an admin can save a configuration with no TLS and no IPSec encryption without any confirmation dialog. For security-critical fields that disable encryption, a confirmation step ("Are you sure? This removes encryption for all connected users") would prevent accidental misconfiguration.

`resetToDefaults` indexes `PROTOCOL_DEFAULTS[proto]` without a guard. If the constant lacks an entry for the active protocol, `defaults` is `undefined` and the destructuring assignments (`defaults.port`, `defaults.network`, etc.) will throw a TypeError.

## Node agent: VPN disconnect fallback is dead code

The `disconnectVPNUser` function's last-resort path finds the CCD file and logs intent:

```go
ccdPath := fmt.Sprintf("/etc/openvpn/server/ccd/%s", username)
if _, err := os.Stat(ccdPath); err == nil {
    log.Info("disabling CCD for user to force disconnect", ...)
}
return "", fmt.Errorf("could not connect to OpenVPN management interface...")
```

No `os.Rename` call follows the log statement. The function returns an error regardless. Either implement the rename (`os.Rename(ccdPath, ccdPath+".disabled")`) and return success, or remove the dangling `os.Stat` check that gives a false impression of fallback behavior.

- `disconnectVPNUser` username validation (injection prevention)
- The bidirectional locale watch (infinite loop prevention)
