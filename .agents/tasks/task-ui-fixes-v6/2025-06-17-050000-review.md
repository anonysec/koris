# Full i18n, JSON backup, crypto payments, Telegram bot expansion, outbound proxy config

This changeset delivers five major feature areas across the Go backend and both Vue frontends: complete i18n coverage with Russian locale added, a JSON-based database backup/restore system replacing the CSV-only export UI, crypto payment method support with wallet fields, an expanded Telegram bot with inline keyboards and new admin commands (/renew /broadcast /online /backup /nodes), and an outbound/tunnel proxy configuration for VPN nodes (vless/vmess/trojan/shadowsocks/socks5). The RTL topbar fix uses physical direction overrides to keep action buttons on the right regardless of reading direction, and a theme visual restyle adds per-theme radius/shadow/font overrides via CSS custom properties.

Watch for: backup import silently discards per-row SQL errors making partial imports invisible (**confirmed**); outbound proxy functions are dead code never called from the node agent (**confirmed**); `user:` callback in the Telegram bot is not admin-guarded, leaking customer details to any user who crafts the callback (**confirmed**); exported nodes and vpn_configs are not reimported, creating asymmetry that will confuse administrators (**confirmed**).

## High-level view

The i18n work covers all admin views with `t()` for labels, buttons, toasts, and validation messages across EN/FA/ZH/RU. A handful of input placeholders remain hardcoded English (e.g. "Password", "Display name", "Plan default") which will appear untranslated in FA/RU deployments.

The RTL fix applies `row-reverse` to the topbar flex container and forces `direction: ltr` on the action cluster, correctly pinning buttons to the physical right regardless of document direction. The portal header uses the same pattern.

The backup system wraps imports in a single transaction with rollback-on-error. However, it exports 9 tables (including `nodes` and `vpn_configs`) but only imports 7, creating a lossy round-trip. Every row-level INSERT failure is swallowed (`_, _ = tx.Exec(...)`) so a half-successful import reports `{"ok": true}` with no indication of which rows failed.

The outbound proxy module generates bridge configs compatible with xray-core/sing-box, but `parseOutboundConfig` and all `applyOutbound*` functions are never invoked anywhere in the node agent — the data flows from frontend to database but never reaches the runtime.

The Telegram bot's callback handler doesn't guard the `user:` prefix behind `isAdmin`, so any customer chat that sends a crafted callback data string can invoke `cmdFind` and retrieve another user's account details, wallet balance, and subscription info.

## Backup export/import transaction safety

The critical gap is error visibility. Every row insert uses `_, _ = tx.Exec(...)` — if a column type mismatch, FK violation, or schema difference causes a row to fail, that error vanishes. Since `INSERT IGNORE` is used for most tables, MySQL will silently skip rows that violate unique constraints without an error, which is intentional for idempotent re-import. But the combination means there's no way to distinguish "imported successfully" from "skipped 80% of rows due to schema mismatch" — both return `{"ok": true}`.

The export queries `vpn_configs` and `nodes`, but the import handler processes only: plans, customers, wallets, radcheck, payments, subscriptions, tickets, wallet_transactions. A backup taken from panel A cannot fully restore node infrastructure on panel B.

## Outbound proxy: complete module, no integration point

The backward-compatibility design is correct — existing nodes without the `outbound` key in `extra_json` will get `nil` from `parseOutboundConfig` and all `apply*` functions return immediately on nil:

```go
func parseOutboundConfig(extraJSON json.RawMessage) *OutboundConfig {
    if len(extraJSON) == 0 { return nil }  // backward-compat: old nodes without extra_json
    ...
    if !cfg.Enabled || cfg.Address == "" { return nil }  // disabled = no-op
    return &cfg
}
```

However, grep confirms no caller exists in the node agent's protocol setup code. The UI in `NodesView.vue` writes the outbound config into `extra_json` and the API validates the type field, so the data flows from frontend to database — it just never reaches the node agent's runtime.

Additionally, `buildOutboundEntry` stores port as a string (`port := "443"`) in the generated JSON, but xray-core/sing-box expect an integer for the port field. This will produce a config that xray rejects when eventually wired in.

## Telegram bot: `user:` callback leak

All new slash commands (`/renew`, `/broadcast`, `/online`, `/backup`, `/nodes`) are properly guarded with `if isAdmin` checks. The callback handler correctly guards `enable:`, `disable:`, `reset:`, and `menu:` admin actions.

The `user:` callback at line 318 is the exception:

```go
if strings.HasPrefix(data, "user:") {
    username := strings.TrimPrefix(data, "user:")
    b.cmdFind(chatID, username)  // no isAdmin check
}
```

`cmdFind` reveals username, display name, status, plan name, wallet balance, data usage, and session count. A non-admin user who discovers this pattern can query any customer's account info.

`cmdNodes` has a misleading SQL subquery that computes total global online users (not per-node) and scans it into a variable (`online`) that is never used in the output format string. Harmless but wasteful — remove the subquery or display the value.

## i18n remaining gaps

The `Locale` type is updated to include `'ru'` and `getPersistedLocale` validates all four values. The language persistence fix correctly prioritizes `localStorage` over server settings, preventing the sidebar selector from being overridden on reload.

Remaining hardcoded strings in templates: `<th>IP</th>` in CustomerDetailView, `placeholder="Password"`, `placeholder="Display name"`, `placeholder="Plan default"` in CustomerDetailView, `placeholder="UUID or password"` in NodesView. The first is arguably universal; the rest will be visible to non-English users as the only English text in an otherwise translated interface.

## Webhook dispatch: no validation, no retry

`dispatchWebhook` launches a goroutine with a 10-second HTTP client timeout. On failure, it logs and returns — no retry, no backoff, no dead-letter. For warning notifications this is low-severity, but the URL itself is never validated at write time. An admin entering `htps://` (typo) or a bare domain will only discover the problem when warnings fail to arrive, with no UI feedback.
