# Ten-fix omnibus: login, bot DB config, proxy redesign, L2TP cleanup, advanced toggle, SSH accounting/status, cert tab, i18n, vite pin

This commit bundles ten distinct changes: removing the preflight health check from admin login (adding an explicit error fallback), reading Telegram bot config from the DB with env-var override, replacing per-protocol xray outbound fields with a per-node HTTP/SOCKS5 API proxy, stripping redundant L2TP auth toggles, hiding security/performance/logging behind an advanced toggle, adding SSH accounting fields, populating SSH status from the services table, converting the settings cert tab to panel HTTPS upload, adding i18n keys across EN/FA/ZH/RU, and pinning vite to ^5.4.19.

**Watch for:** The proxy fields sent from the frontend have no backend persistence — the `updateNode` handler only accepts name/public_ip/domain (confirmed). The cert upload UI calls `/api/settings/cert-status` and `/api/settings/cert-upload`, but neither endpoint exists in the Go router (confirmed). The hardcoded English fallback error `"Invalid username or password"` in the auth store is not i18n-aware (confirmed).

## High-level view

The login fix removes a no-op preflight `GET /api/health` and adds a catch branch that sets `error.value` to a friendly string when the API throws. The fallback string is English-only, bypassing the i18n system that the same commit extends with new keys in four languages.

The bot startup now loads `telegram_token` and `telegram_chat_id` from `panel_settings` before checking environment variables, auto-enabling when a token is present and `PANEL_TG_ENABLED` is unset. The env-var-wins precedence is explicit and the query gracefully tolerates a missing table.

The proxy redesign removes the per-protocol `outbound` object (vless/vmess/trojan/socks5 bridge config) from `PROTOCOL_DEFAULTS` and replaces it with per-node `proxy_enabled/proxy_type/proxy_address/proxy_username/proxy_password` in the edit-node form. The Go backend's `updateNode` handler still only accepts `{name, public_ip, domain}` — the proxy fields are silently discarded on save. The node agent's `outbound.go` (which implements the old xray-bridge approach) is now unreachable from the UI but still present in the codebase.

The L2TP cleanup removes `refuse_chap`, `refuse_pap`, and `require_mschap_v2` from the form and defaults — redundant with the `auth_method` select. Existing `extra_json` rows carry these keys harmlessly; they'll vanish on next save.

The cert tab UI calls two endpoints that don't exist in the backend. `loadCertStatus()` silently swallows the 404, but `saveCertFiles()` will show a generic error toast. The upload function also bypasses `useApi`, skipping CSRF token injection.

SSH accounting adds `accounting_enabled` and `accounting_interval` to the SSH defaults in `extra_json`, but no node-agent code consumes these fields yet.

## Proxy redesign: frontend without backend

The edit-node form now sends a payload like `{name, public_ip, domain, proxy_enabled: true, proxy_type: "SOCKS5", proxy_address: "1.2.3.4:1080", proxy_username: "u", proxy_password: "p"}` via `PATCH /api/nodes/:id`. The handler at `api.go:1877` decodes into:

```go
var in struct {
    Name     string `json:"name"`
    PublicIP string `json:"public_ip"`
    Domain   string `json:"domain"`
}
```

The five proxy fields are silently discarded. There is no database column (`proxy_enabled`, etc.) on the `nodes` table, so even if the struct accepted them, there's no persistence. The `EditNodePayload` TypeScript interface in `nodes.ts` also only declares `name?/public_ip?/domain?`, so the type system offers no protection at the call site (the payload is cast to `Record<string, any>`).

The old `outbound` object that lived inside `extra_json` per protocol config was functional — the node agent's `outbound.go` parses it and configures xray/sing-box bridges. The new per-node proxy is conceptually different (API-level proxy for the node agent's HTTP client to reach the panel), but neither the agent nor the panel backend has been updated to implement it.

## Cert tab: UI calls nonexistent endpoints

`saveCertFiles()` POSTs a multipart form to `/api/settings/cert-upload`:

```typescript
const res = await fetch('/api/settings/cert-upload', {
  method: 'POST',
  credentials: 'include',
  body: formData,
})
```

This skips CSRF token injection (which `useApi` handles via the `X-CSRF-Token` header). If the backend enforces CSRF on POST routes — and it likely does given the infrastructure — the upload will be rejected even once the endpoint exists. The fix is to either use `useApi` with a custom content-type flow, or manually inject the CSRF token from the module-level `getCsrfToken()` export.

## Login error message hardcoded in English

The catch block in `login()`:

```typescript
} catch {
  if (!error.value) {
    error.value = 'Invalid username or password'
  } else if (error.value === 'Unauthorized') {
    error.value = 'Invalid username or password'
  }
  return false
}
```

Users with locale set to FA/ZH/RU will see English text in the error banner. The fix: add a key like `'login.invalid_credentials'` to all locales and use the i18n translation function. Since the store doesn't have access to `t()` directly, either pass the translated string from the calling component or import the i18n instance at module level.
