# Dead code removal and shared utility consolidation (v2)

This change removes ~2400 lines of dead code (a monolithic legacy SFC, orphaned test files, unused utility modules) and consolidates duplicated date formatting and stale-data logic into the shared library. A documentation file catalogs shared component/composable usage across apps.

The v2 pass addresses all three issues from the prior review: 13 unit tests now cover `useFormatDate`, the `COMPONENT_USAGE.md` clarifies `useFreshData` is portal-only (admin uses WebSocket realtime), and a TODO comment marks admin date formatting as a future migration target. JSDoc on `useFormatDate.ts` explains that the `use` prefix is retained for import consistency despite the module exporting plain functions.

**Watch for:** `formatDate` silently returns a locale-dependent string for invalid date inputs like `"not-a-date"` (produces "Invalid Date" behavior varies by runtime) — no validation beyond the falsy check (likely). Admin still formats dates with `.slice(0, 10)` in 6 views — intentionally deferred via TODO but creates visual inconsistency between admin and portal (confirmed).

**Verdict**: APPROVED

## High-level view

The dead code deletion is clean — grep confirms zero dangling references. The `useFormatDate` utility is well-tested (13 cases) and correctly integrated into three portal consumers. The `useFreshData` consolidation is correctly scoped to portal-only, with admin's WebSocket realtime strategy documented as the reason it's unnecessary there.

The one remaining gap is `formatDate`'s behavior on malformed (non-empty, non-null) date strings — `new Date("garbage")` produces `NaN` timestamp, and `Intl.DateTimeFormat.format(NaN)` output is implementation-defined. In practice the panel only feeds ISO strings from the API, so this is a defensive concern rather than a live bug.

## Invalid date passthrough in formatDate/formatDateTime

Both functions guard against null/undefined/empty-string via `if (!value) return fallback`, but a non-empty malformed string like `"pending"` or `"N/A"` would pass this check. `new Date("pending")` yields an Invalid Date object, and `Intl.DateTimeFormat('en', ...).format(invalidDate)` produces runtime-dependent output — Chrome emits `"Invalid Date"`, Node may throw in some versions. A one-line addition:

```ts
const d = new Date(value)
if (Number.isNaN(d.getTime())) return fallback
```

would make the function safe against any string input. Since admin views could eventually migrate to this utility (per the TODO), hardening it now prevents edge-case bugs later. If the guard is added, a corresponding test case for malformed input should accompany it.
