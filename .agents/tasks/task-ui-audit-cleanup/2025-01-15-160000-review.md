# Dead code removal and shared utility consolidation

This change removes ~2400 lines of dead code (a monolithic legacy SFC, orphaned test files, unused utility modules) and consolidates duplicated logic — `useFreshData` and date formatting — into the shared library. It also adds documentation cataloging every shared component and composable with its usage status.

The approach is straightforward: delete unreferenced files, promote duplicated code to `@koris/composables`, rewire portal imports, and verify builds + tests pass. Net deletion is over 2400 lines with under 100 lines added.

**Watch for:** No unit tests for the new `useFormatDate` utility (confirmed). Admin views still use `?.slice(0, 10)` for date display, leaving the consolidation incomplete (confirmed). The `useFreshData` composable is documented as used in portal only, but the admin app no longer uses it at all — raising the question of whether the consolidation achieves its stated goal of sharing across apps (confirmed).

## High-level view

The dead code removal is safe — grep confirms zero dangling imports after the deletions. The 2142-line `App.legacy.vue` was truly orphaned (the current `App.vue` just renders `<router-view />`).

The `useFreshData` move from admin to shared is technically correct (identical copies existed in both apps), but the admin app no longer imports it from anywhere. The composable is only consumed by two portal views. Whether the admin app intentionally stopped using stale-data guards or accidentally lost them during an earlier refactor is worth confirming.

Admin views still format dates with raw `.slice(0, 10)` string truncation in at least five views (CustomersView, ResellersView, DashboardView, CustomerDetailView, PaymentsView). The `useFormatDate` consolidation only covers the portal half.

## Incomplete date formatting consolidation

At least five admin views render dates via `?.slice(0, 10)`:

```
{{ tx.created_at?.slice(0, 10) }}        // CustomerDetailView
{{ sub.start_date?.slice(0, 10) }}       // CustomerDetailView  
{{ value?.slice(0, 10) }}                // ResellersView (x2)
{{ value?.slice(0, 10) }}                // CustomersView
{{ user.created_at?.slice(0, 10) }}      // DashboardView
```

This pattern is fragile — it assumes ISO 8601 ordering and produces raw `2025-01-15` output without locale formatting. The shared `formatDate` would produce `Jan 15, 2025` and gracefully handle null/undefined inputs with a `'--'` fallback. Leaving admin unconsolidated means the two apps display dates in visually different formats for the same underlying data.

## Missing test coverage for useFormatDate

`useFreshData.test.ts` was correctly moved alongside its source and passes its 6 tests. But `useFormatDate.ts` has no corresponding test file. The function has branching logic (null check to fallback, otherwise format via `Intl.DateTimeFormat`) and a configurable `fallback` parameter — both should have coverage. The shared test infrastructure already has the alias registered in vitest.config.ts, so adding a test file is trivial.

## useFreshData scope mismatch

The composable implements a stale-data guard: on mount and on keep-alive reactivation, it re-fetches if more than 30 seconds have passed. After this change, only two portal views (`BillingView`, `UsageView`) consume it. The admin app's data-loading is handled entirely through its Pinia stores with WebSocket-driven realtime updates, so it may have legitimately outgrown this pattern. The COMPONENT_USAGE.md should reflect this — currently it says "consolidated from duplicates in admin and portal" without noting that admin no longer uses it.

## Documentation naming inconsistency

The COMPONENT_USAGE.md lists `useFormatDate` as a composable, but the module exports plain functions (`formatDate`, `formatDateTime`) rather than following the Vue composable pattern (no `ref`, no lifecycle hooks, no reactive state). The file could live at `shared/utils/formatDate.ts` instead. Not blocking, but creates a naming precedent where non-reactive utilities get the `use` prefix alongside actual composables like `useFreshData` (which uses `onMounted` and `onActivated`).
