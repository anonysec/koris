# Admin panel overhaul: notifications, i18n, protocol config, portal simplification, and node agent hardening

A broad set of changes across the VPN panel admin UI, customer portal, and the Go node agent. The admin panel gains a real-time notification system (WebSocket-pushed from the backend, surfaced via bell dropdown and a dedicated page), comprehensive i18n coverage (~130 keys across EN/FA/ZH), expanded VPN protocol configuration with hint tooltips, a compact customer table, and reseller improvements. The customer portal is simplified from 4 to 3 navigation tabs with a mobile bottom-tab-bar. On the backend, the node agent adds graceful shutdown, three new task actions (service.stop, agent.update, vpn.disconnect-user), enhanced service detection (dropbear, L2TP/IPSec compound status), IPv6 fallback in `firstIP()`, and a race condition fix in `loadConfigFromEnv`.

**Watch for:** Command injection in `disconnectVPNUser` where an unsanitized username is written directly to the OpenVPN management socket (confirmed). Notification state is ephemeral — stored only in browser memory with no persistence or API fetch on page load, so notifications vanish on refresh (confirmed). The `defer conn.Close()` inside a loop in `disconnectVPNUser` accumulates deferred closes until function exit rather than closing each failed connection immediately (confirmed).

## High-level view

The notification pipeline is wired end-to-end: the Go backend broadcasts a WebSocket message when a customer creates a support ticket, the admin frontend's realtime store captures it, and the topbar bell renders a hover dropdown with a "view all" link to a new `/notifications` page. Notifications are purely ephemeral — no server-side persistence, no fetch-on-load, no deduplication — so any missed WebSocket frame or page refresh means lost notifications.

The node agent's new `vpn.disconnect-user` task writes an attacker-controlled username directly into the OpenVPN management protocol without sanitization. The management interface interprets newline-delimited commands, so a username containing `\n` followed by another management command could alter server state beyond disconnection.

The i18n expansion covers all three locales symmetrically, and the spread-operator config merging in the protocol form ensures saved configs retain existing keys when new defaults are added — an important upgrade-friendliness property.

The portal simplification removes the Billing tab, consolidates to 3 tabs (Dashboard, Support, My VPN), and adds a mobile bottom tab bar with `env(safe-area-inset-bottom)` for notch-friendly rendering. The portal nav links are hardcoded in English, unlike the admin panel which uses `t()` throughout.

## Username injection via OpenVPN management protocol

The `disconnectVPNUser` function constructs a kill command by interpolating the username directly:

```go
killCmd := fmt.Sprintf("kill %s\n", username)
conn.Write([]byte(killCmd))
```

The `username` parameter arrives from a task payload (`fmt.Sprint(payload["username"])`). While there is a nil/empty check, there is no validation that the username is free of newlines or other control characters. The OpenVPN management interface processes commands line-by-line, so a payload like `{"username": "alice\nstatus\n"}` would execute the `status` command after the kill. More destructive commands (e.g. `signal SIGTERM`) are also on the management interface.

The only upstream check is `username == "" || username == "<nil>"`, which doesn't cover injection. Add a validation step that rejects usernames containing `\n`, `\r`, or non-printable characters. (confirmed)

## Connection leak pattern in disconnectVPNUser

```go
for _, sockPath := range mgmtPaths {
    conn, err := net.DialTimeout("unix", sockPath, 3*time.Second)
    if err != nil { continue }
    defer conn.Close()
    // ... use conn, return on success
}
```

If the first socket connects but the write fails (the `continue` path after the write error), the loop moves to the next socket. The first connection remains open until the function exits. For two sockets in the list plus the TCP fallback, up to three connections could accumulate before any deferred close runs. Extract the per-socket logic into a helper or use explicit `conn.Close()` on the error path. (confirmed)

## Notification lifecycle gap

Notifications exist only as a Vue `ref<Notification[]>` populated by WebSocket `"notification"` frames. There is no API endpoint to fetch historical notifications, no persistence to localStorage, and `markAllRead` mutates only the local array. Consequences:

- Page refresh: all notifications gone.
- Multiple browser tabs: read state inconsistent.
- WebSocket reconnect after disconnect: missed notifications are permanently lost.

The server-side `broadcastNotification` fires and forgets — the notification map is never stored in the database. The admin could miss a support ticket if they weren't connected at the exact moment it arrived. (confirmed)

## Untranslated strings in Settings and Portal

In `SettingsView.vue`, the toast messages use raw English:

```typescript
toast.success('Panel settings saved successfully.')
toast.error('Failed to save panel settings.')
toast.success('Telegram settings saved successfully.')
```

These should use `t('settings.save_success')` etc. to maintain full i18n coverage. (confirmed)

The portal shell (`PortalShell.vue`) navigation links — "Dashboard", "Support", "My VPN" — are hardcoded strings in both the desktop `<nav>`, the mobile overlay, and the bottom tab bar. The portal has no `useI18n` import at all. (confirmed)

## Silent notification drop under load

The `broadcastNotification` in `api.go` uses a non-blocking send to each subscriber channel (buffer size 16). If a subscriber's buffer fills — due to a slow WebSocket client or a burst of tickets — notifications are silently dropped for that subscriber with no retry or logging. For a panel with typically 1-2 admin sessions this is unlikely to trigger, but the fail-silent behavior means dropped notifications are invisible to both operator and user. (likely)
